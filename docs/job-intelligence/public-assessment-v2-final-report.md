## 4. Competency, dimension and Evidence Signal coverage

Mapped onto the existing 14-`DimensionId` legacy model (deliberate scoping decision, see §12 below), not Assessment DNA's new 12-Dimension model. Full per-question mapping in the authoring spec. Coverage summary: `structure_documentation`, `risk_awareness`, `conflict_management`, `communication`, `independent_decision_making` each receive 2–4 methodologically distinct items; `teamwork`, `learning_development`, `service_orientation`, `analytical_orientation`, `operational_orientation` receive 2 items; `leadership_orientation`, `strategic_orientation`, `technical_orientation`, `investigation_orientation` receive option-level-only coverage via Q16 (reported limitation, not hidden).

## 5. Response formats used and why

Single-select (behavioural/SJT/dilemma framing) for 6 items, Likert 1–5 for 8 items, one deliberate 1–10 confidence item (Q11, per Measurement Framework §2's "communication fidelity" exception — not a default), and one multi-select (Q16). No item defaults to a universal scale. Full reasoning per item in the authoring spec.

## 6. Candidate result structure

Unchanged engine and UI (`EngineResultV1` → `EngineResultView`), verified live against the new questions: career profile (archetypes/motivations/working style), family ranking, top profession matches with confidence tiers, strengths ("areas that stood out"), development areas ("where evidence is weaker" — explicitly framed as "not a judgement of your ability"), formal requirements, education/certification pathways, action plan, and — already present, verified unchanged — an explicit guidance-not-eligibility/hiring-decision disclaimer in both languages.

## 7. Employer report structure

New `EmployerReportView` component (admin-only preview route today, see §12): summary (top match, confidence, overall evidence level), competency evidence with **evidence-type labels** (self-report / scenario-based / behavioural / preference) per strength and development area, explicit "this is inferred guidance, not a verified deficiency" framing on every development item, deterministic templated interview-focus suggestions per weak dimension, a "requires verification by employer" section sourced from the engine's existing formal-requirements enrichment, and a prominent human-decision disclaimer stating the report never approves, rejects, ranks, or shortlists.

## 8. Scoring and versioning behaviour

Deterministic only — zero LLM/AI involvement anywhere in the scoring or explanation pipeline (verified: unchanged `matching-engine.ts`/`scoring.ts`, no new AI dependency introduced). Versioning: one new additive `assessment_versions` row (`model_version = '2026.07-v2.0'`) resolved automatically by the existing "latest published version" query in `saveMyCareerReport()` — new completions attribute to v2.0, every existing completed run keeps its already-stored, immutable `assessment_version_id` untouched. Reproducibility: a completed report is an immutable JSON snapshot independent of future code changes (unchanged architecture, verified by reading `report-types.ts`).
